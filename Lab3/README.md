# Лабораторная работа 3: Управление заказами в службе доставки

> Разработайте систему для управления заказами в службе доставки еды.

Система должна:

1. Поддерживать создание и управление заказами, включая возможность выбора блюд из меню.
2. Обрабатывать различные типы заказов, такие как стандартные заказы и заказы с особенными условиями (например, быстрая доставка, персональные предпочтения).
3. Предоставлять возможность отслеживания состояния заказа: подготовка, доставка, выполнен.
4. Включать функции для расчета стоимости заказа с учетом различных факторов, таких как скидки, налоги, плата за доставку.
5. Логику необходимо покрыть unit тестами
6. CLI не нужен
7. Использовать не менее 5 паттернов ООП, изученных за курс. Иметь четкое обоснование почему был выбран каждый

---

## Примененные паттерны проектирования
1. Фабрика (Factory Method)
   - Зачем использовал: Для создания различных типов заказов (стандартные, экспресс-заказы) без жесткой привязки клиентского кода к конкретным классам
   -Польза: Упрощает добавление новых типов заказов

2. Стратегия (Strategy)
   - Зачем использовал: Для реализации различных алгоритмов расчета стоимости заказа (стандартный, со скидкой, для лояльных клиентов).
   - Польза: Позволяет динамически менять логику расчета цены, легко добавлять новые стратегии ценообразования

3. Состояние (State)
   - Зачем использовал: Для управления жизненным циклом заказа и обработки переходов между состояниями (новый, подтвержден, готовится, доставляется, выполнен, отменен)
   - Польза: Чтобы избавиться от ифов

4. Наблюдатель (Observer)
   - Зачем использовал: Для реализации системы уведомлений о изменениях статуса заказа (клиент, кухня, служба доставки).
   - Польза: Обеспечивает слабую связанность между компонентами, позволяет легко добавлять новых подписчиков

5. Декоратор (Decorator)
   - Зачем использовал: Для динамического добавления дополнительных опций к заказу (подарочная упаковка, срочный заказ).
   - Польза: Позволяет добавлять новую логику без новых классов

6. Команда (Command)
   - Зачем использовал: Для инкапсуляции операций с заказами (создание, изменение статуса, отмена) и реализации механизма отмены/повтора.
   - Польза: История, отмена операций
